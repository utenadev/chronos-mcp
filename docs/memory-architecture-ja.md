# 記憶システム設計書：種別とライフサイクル

## 1. 記憶の4層構造 (Memory Taxonomy)

本システムでは、人間の脳科学的プロセスを模倣し、記憶を以下の4つのレイヤーで管理します。

### 第1層：短期記憶 (Simple Memory)
- **データ実体**: `conversation_turns` テーブル
- **役割**: 会話の生ログ（発言ペア）。
- **取り扱い**: 時系列でそのまま記録。直近の文脈（コンテキスト）として利用されます。
- **対応状況**: **【実装完了】** Goの `record_turn` ツールにより自動記録。

### 第2層：関連性記憶 (Relational Memory / GraphRAG)
- **データ実体**: `memory_snapshots` ( `causality_id` を使用 )
- **役割**: 知識間の因果関係（「なぜその結論に至ったか」）の保持。
- **取り扱い**: スナップショット間をIDで紐付けることで、思考のネットワークを構築します。
- **対応状況**: **【基盤完了】** Go側での紐付け用フィールドの実装。自動抽出ロジックは Python 側で実装中。

### 第3層：ペルソナ・アンカー (Persona Anchors) ★独自機能
- **データ実体**: `memory_snapshots` ( `is_persona_anchor` フラグを使用 )
- **役割**: ユーザーの嗜好、価値観、固有の話し方など「共感」の核となる情報。
- **取り扱い**: 通常の整理プロセス（抽象化）では消されがちな「ユーザーの生の声」をあえてそのまま、あるいは高解像度で長期保存します。
- **対応状況**: **【基盤完了】** Go側での識別フラグの実装。抽出ロジックは Python 側で実装中。

### 第4層：統合・再編層 (Consolidation Layer / Wisdom)
- **データ実体**: `memory_snapshots` ( `status_consolidated`, `importance_score` )
- **役割**: 過去の記憶を抽象化し、再利用可能な「教訓」や「ルール」へ変換したもの。
- **取り扱い**: 睡眠サイクル（バッチ処理）によって生成。重要度（スコア）により想起の優先順位が決まります。
- **対応状況**: **【基盤完了】** Go側での整理ステータス管理。要約・抽象化アルゴリズムは Python 側で実装中。

## 2. ライフサイクル：睡眠サイクル (Consolidation Process)

記憶は以下のプロセスを経て「データ」から「知恵」へと昇華されます。

1.  **覚醒フェーズ**: 会話が `conversation_turns` に蓄積され、一時的なスナップショットが作成される（ステータス=未整理）。
2.  **休止フェーズ**: セッション終了イベント (`record_session_event`) がトリガーとなる。
3.  **睡眠フェーズ (バッチ処理)**: 深夜時間帯に Python スクリプトが未整理の記憶を走査。
    -   **海馬リプレイ**: その日の会話を再走査し、サマリを作成。
    -   **インサイト抽出**: 因果関係の特定とペルソナ情報の抽出。
    -   **更新**: スナップショットに重要度を付与し、整理済みフラグを立てる。
4.  **夢フェーズ**: 新しい知見と既存の長期記憶をランダムに突き合わせ、矛盾の検知や伏線の回収（Open Loops）を試みる。

# 海馬リプレイ（記憶整理バッチ）詳細仕様・連携記録

## 1. 概要
「海馬リプレイ」は、エージェントの短期記憶（会話ログ）を長期記憶（構造化された知恵）へと統合するためのバッチ処理プロセスです。脳科学における睡眠中の記憶整理プロセスを模倣しています。

## 2. OpenCode への指示内容と設計合意
Architect (Gemini CLI) と Implementer (OpenCode / Sisyphus) の間で、以下の高度な機能拡張が合意されました。

### 2.1 セッション ID の自律推論
- **課題**: `environment` フィールド（例: "default"）だけでは、過去の別セッションのログが混ざるリスクがある。
- **指示**: 最新の `session_logs` テーブルを参照し、現在どのセッションが進行中（または直近で終了したか）を特定するロジックを組み込むこと。
- **目的**: 記憶の整理を「意味のあるひとまとまり（セッション）」単位で正確に行う。

### 2.2 ペルソナ・アンカーの保護
- **課題**: 通常の要約プロセスでは、ユーザーの細かい好みや話し方が「ノイズ」として削ぎ落とされる可能性がある。
- **指示**: `is_persona_anchor=1` のフラグが立っているスナップショットは、共感のための核心情報として以下の特別扱いを行う。
    - **重要度 (importance_score)**: 強制的に最大値 `1.0` を付与。
    - **要約プロンプト**: LLM に対し「生データを最大限維持し、抽象化しすぎないこと」を明示的に指示。
- **目的**: ユーザーへの「深い理解（Deep Understanding）」を長期的に維持する。

### 2.3 因果関係のグラフ化 (GraphRAG への布石)
- **課題**: 単なるタグ名としての `causality_id` では、複雑な知識の繋がりを表現できない。
- **指示**: `causality_id` フィールドに JSON 構造（案B）を採用する。
    - **形式**: `{"parent_id": 123, "pattern": "error-resolution", "type": "causal"}`
- **目的**: 記憶を「点」ではなく「線（グラフ）」として保持し、将来的に GraphRAG 等で高度な推論を可能にする。

## 3. 実装アプローチ
OpenCode は以下の構成で Python スクリプトを実装しました。

- **`scripts/hippocampal_replay.py`**:
    - `HippocampalReplay` クラスによるオブジェクト指向設計。
    - `dry-run` モードによる安全なシミュレーション機能。
    - キーワードベースのパターン抽出（将来の LLM API 統合を容易にするプラグイン構造）。
- **`scripts/setup_cron.sh`**:
    - 毎日 AM 3:00 に自動実行するためのデプロイ補助。

## 4. 検証・確認方法
実装の正しさは以下の手順で確認されました。

1.  **静的解析**: Go 側の DB スキーマとの整合性確認。
2.  **ユニットテスト**: `scripts/test_hippocampal_replay.py` の実行。
    - 未統合データの抽出テスト。
    - 統合フラグの更新テスト。
    - 因果関係パターンのマッチングテスト。
3.  **統合テスト (Dry Run)**:
    ```bash
    python3 scripts/hippocampal_replay.py --dry-run
    ```
    実際の DB を書き換えずに、どのスナップショットがどのように更新されるかを確認。
